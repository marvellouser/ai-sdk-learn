# 复杂功能交付

## 概述

用这个工作流交付复杂功能：先沉淀需求文档和测试用例，再实施功能，并且只在计划实施完成后的首次交付时自动校验测试用例是否通过。

## 工作流

1. 判断是否适用。
   - 将用户可见的新功能、新工作流、非平凡 UI/API/数据改动、多文件实施计划视为复杂功能。
   - 小修小补、文案调整、依赖升级、单行 bug 修复不要强行套用，除非用户明确要求需求文档或测试用例。

2. 先理解仓库。
   - 创建文件前，先查看项目结构、文档目录约定、测试脚本、已有实现模式。
   - 优先沿用仓库现有文档目录、命名风格和测试方式。

3. 创建或更新需求包。
   - 如果仓库使用 `docs/`，优先放在 `docs/requirements/<feature-slug>/`。
   - 默认包含 `requirements.md` 和 `test-cases.md`。
   - `<feature-slug>` 使用简短的小写中划线命名。
   - 如果仓库已有其他约定，跟随已有约定，但需求文档和测试用例必须放在同一个功能目录下。

4. 编写 `requirements.md`。
   - 包含背景、目标、用户场景、功能范围、输入输出、业务规则、校验规则、边界情况和默认值。
   - 文档要能指导实现，但不要过度绑定私有代码符号。
   - 重要假设必须显式记录。

5. 编写 `test-cases.md`。
   - 用表格导出测试用例，字段包含：用例编号、模块、场景、输入、操作步骤、预期结果、优先级。
   - 覆盖主流程、边界场景、校验失败、导航/集成点和用户指定的业务规则。
   - 用例要具体到测试人员或自动化测试作者不需要猜。

6. 实施功能。
   - 尽量添加或更新能映射到 `test-cases.md` 的自动化测试。
   - 易测的业务逻辑放到 helper/lib 模块，UI 组件聚焦状态和展示。
   - 同步更新路由、导航、首页入口、索引、文档链接或功能注册表等可发现入口。

7. 计划实施完成后的自动校验。
   - 仅在该计划首次实施完成时，自动运行与测试用例对应的测试、类型检查、lint、构建或页面 smoke check。
   - 前端功能如果能启动 dev server 或静态访问页面，需要做一次路由/页面 smoke check。
   - 如果部分测试用例只能人工验证，说明已自动覆盖的部分和仍需人工验收的部分。
   - 如果某个校验命令无法运行，说明原因和残余风险。

8. 后续追加修改的测试规则。
   - 计划已经实施并完成首次自动校验后，后续如果用户继续要求改文案、调规则、补字段、修细节或改实现，不要默认再次运行测试。
   - 只有当用户主动提到“测试”“验证”“跑一下”“检查是否通过”“再确认”等明确意图时，才再次运行测试或校验命令。
   - 如果按本规则没有再次运行测试，最终回复中简短说明“此次未重新运行测试，因为用户未要求再次校验”。

9. 最终回复。
   - 总结新增或更新的需求文档、测试用例、实现范围和首次校验结果。
   - 给出最有用的本地页面 URL 或文件路径。
   - 只在重要时说明假设、未运行的检查或后续风险。
