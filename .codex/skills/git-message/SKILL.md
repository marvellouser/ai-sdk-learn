# Git Message

## 生成流程

1. 优先基于 staged changes 生成提交信息：
   - 先查看 `git diff --cached --stat`。
   - 如需细节，再查看 `git diff --cached`。
2. 如果没有 staged changes，但用户仍要求生成提交信息：
   - 查看 `git diff --stat` 和必要的 `git diff`。
   - 在回复中说明“当前没有 staged changes，以下基于工作区改动生成”。
3. 不要编造未出现在 diff 中的改动。
4. 除非用户明确要求，不要自动执行 `git add`、`git commit` 或 `git push`。

## 提交类型

从以下类型中选择一个最贴切的类型：

- `feat`：新增功能或用户可见能力。
- `fix`：修复 bug、错误处理或异常行为。
- `docs`：文档、说明、示例或注释更新。
- `test`：新增或调整测试。
- `refactor`：不改变行为的代码重构。
- `style`：格式、样式、排版等不影响逻辑的改动。
- `perf`：性能优化。
- `build`：构建、依赖、脚本或打包配置变更。
- `ci`：CI/CD 配置变更。
- `chore`：维护性改动、杂项调整。
- `revert`：回滚提交。

## 输出格式

提交信息必须使用中文，并包含提交类型、提交详细内容、模型名称。

默认输出一个可直接作为 commit message 使用的多行文本：

```text
<type>: <一句中文摘要>

提交类型：<type>
提交详细内容：
- <具体改动 1>
- <具体改动 2>
- <具体改动 3>

模型名称：<当前模型名称>
```

## 编写规则

- 第一行使用 `<type>: <一句中文摘要>`，摘要不超过 72 个中文字符。
- `提交详细内容` 使用 2 到 5 条中文 bullet，只覆盖核心文件、行为变化和必要的兼容/迁移说明。
- 不要把测试、构建、lint 等验证命令或“验证通过”结果写入提交信息。
- 如果用户需要验证结果说明，只能在提交信息外单独补充，不能作为 commit message 的一部分。
- `模型名称` 写当前会话实际模型名；如果运行环境没有显式暴露模型名，写 `Codex`。
- 只输出提交信息本身；除非用户要求解释，不要附加额外说明。
